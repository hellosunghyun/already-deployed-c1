# 00. 워크북 사용법

## 이 문서는 어떻게 읽나요?

이 워크북은 일반 강의 노트가 아닙니다. 각 파일은 **하나의 실습 단계**입니다.

각 단계는 아래 구조를 가집니다.

- **목표**: 이 단계가 끝나면 무엇이 생기는가
- **왜 하는가**: 이걸 하지 않으면 다음 단계가 왜 어려운가
- **쉬운 설명**: 처음 보는 단어를 짧게 풀어 설명
- **AI에게 보낼 프롬프트**: 그대로 복사해서 사용 가능
- **체크박스**: 내가 실제로 했는지 확인
- **Ready Gate**: 다음 단계로 넘어가도 되는지 확인
- **힌트**: 바로 정답을 주지 않고 단계적으로 도움
- **퀴즈**: 헷갈릴 수 있는 개념 확인
- **저장 체크포인트**: 어떤 상태에서 저장점을 남기면 좋은지 안내

## 2회차에서 특히 중요한 규칙

## 오늘의 속도 기준

2회차 실습은 90분을 기준으로 합니다.
모든 문장을 완벽히 다듬는 수업이 아니라, 방문자가 움직일 수 있는 구조를 만드는 수업입니다.

```text
0-15분: 출발 상태 확인
15-35분: 페이지 만들기
35-55분: Layout과 Navigation 만들기
55-75분: Home / About 내용 정리
75-85분: Projects / Posts 기본 구조 만들기
85-90분: build 결과와 다음 질문 기록하기
```

중간에 늦어져도 실패가 아닙니다.
다만 70분이 지났는데 아직 페이지가 하나도 열리지 않는다면, 디자인을 멈추고 `/about`, `/projects`, `/posts`가 열리는지부터 확인합니다.

### 1. 새로 만들기보다 이어서 작업합니다

나쁜 요청:

```text
홈페이지 새로 만들어줘.
```

좋은 요청:

```text
1회차에서 만든 Astro 개인 홈페이지를 유지하면서
Home / About / Projects / Posts 구조를 추가하고 싶어.
기존 배포 설정은 망가뜨리지 말고, 먼저 계획만 설명해줘.
```

### 2. 페이지와 섹션을 구분합니다

오늘 만들 것은 주소가 따로 있는 페이지입니다.

```text
/about
/projects
/posts
```

Home 안에 작은 구역을 만드는 것과 다릅니다.

### 3. 파일 수정 전 계획을 받습니다

2회차는 수정 파일이 많아집니다. 항상 이렇게 시작하세요.

```text
아직 파일을 수정하지 말고, 먼저 계획을 설명해줘.
내가 승인하면 진행해줘.
```

### 4. commit은 저장 체크포인트에서만 다룹니다

`AGENTS.md`에 commit 규칙이 들어 있으므로, 각 실습 프롬프트마다 commit을 요청하지는 않습니다.
워크북에서는 **저장하면 좋은 시점**과 **사용할 수 있는 commit 메시지 예시**만 안내합니다.

오늘 권장 저장 단위는 아래와 같습니다.

```text
feat: 페이지 구조 추가
feat: 공통 레이아웃과 내비게이션 추가
feat: 홈과 소개 페이지 정리
feat: 프로젝트 카드와 글 목록 추가
```

실제 commit은 변경사항을 확인한 뒤, 10단계에서 push까지 정리하거나 사용자가 명시적으로 승인했을 때 진행합니다.

## 막혔을 때 기본 질문 템플릿

```text
나는 2회차 워크북의 [단계 이름] 단계에 있어.
목표는 [원하는 결과]야.
현재 프로젝트는 Astro이고, GitHub Pages에 배포할 예정이야.

현재 상태:
- 내가 방금 한 일: [입력]
- 실행한 명령어: [입력]
- 나온 오류 또는 이상한 화면: [입력]

제약:
- 새 라이브러리는 추가하지 마.
- 관련 없는 파일은 수정하지 마.
- 먼저 원인을 설명하고, 다음 한 단계만 제안해줘.
```
