# 08. Posts 기본 구조 만들기

## 목표

Posts 페이지에 샘플 글 목록을 만듭니다.

## 왜 하는가

나중에는 글 데이터를 다른 콘텐츠 소스에서 가져오게 될 수 있습니다.
하지만 그 전에, 글 목록이 홈페이지에서 어떤 구조로 보일지 먼저 정해두면 이후 작업이 쉬워집니다.

## 쉬운 설명

오늘 Posts는 실제 블로그 시스템이 아닙니다.
Posts를 Projects와 따로 두는 이유는 콘텐츠의 모양이 다르기 때문입니다.
Projects는 비슷한 형식의 카드와 상세정보가 반복되는 구조이고, Posts는 글이 중심입니다.

오늘은 샘플 데이터로 아래 형태를 만듭니다.

이 항목은 이번 수업에서 글 목록을 이해하기 위한 기본 기준입니다.
나중에 본인 홈페이지 성격에 맞게 카테고리, 링크, 외부 글 모음, 짧은 메모 형태로 바꿔도 됩니다.

- 글 제목
- 글 주소용 이름
- 날짜
- 요약
- 태그

오늘은 글 정보를 `src/data/posts.json` 같은 JSON 파일에 따로 둡니다.
나중에 다른 콘텐츠 소스나 MDX 같은 글 파일 방식으로 바꾸더라도 화면 구조를 크게 흔들지 않기 위해서입니다.
글 주소용 이름은 `slug`라고 부릅니다.
오늘은 글 상세 페이지를 만들지 않더라도, 나중에 `/posts/first-deploy` 같은 주소를 만들 때 사용할 수 있습니다.

## 아주 쉽게 말하면

오늘 Posts는 완성된 블로그가 아닙니다.

오늘 만드는 것은 글이 들어갈 "진열대"입니다.

```text
지금: src/data/posts.json에 샘플 글 1~2개를 적어둠
나중: 다른 콘텐츠 소스나 MDX 같은 글 데이터로 바꿔 같은 자리에 보여줌
```

그래서 오늘은 글 상세 페이지, 검색, 댓글, 외부 API를 만들지 않습니다.
목표는 **글 목록이 사이트 안에서 어떤 모양으로 보일지** 먼저 정하는 것입니다.

## SwiftUI와 비교해서 이해하기

SwiftUI에서 글 목록을 만든다면 보통 여러 글 데이터를 목록으로 두고 `List`나 `ForEach`로 보여주는 방식을 떠올릴 수 있습니다.

| Astro Posts | SwiftUI에서 떠올릴 쉬운 개념 |
|---|---|
| `src/data/posts.json` | 글 목록 데이터 |
| 글 카드 또는 목록 항목 | 글 하나를 보여주는 작은 View |
| `title`, `date`, `summary`, `tags` | 글 데이터의 속성 |
| 나중에 다른 콘텐츠 소스로 교체 | 같은 화면에 다른 데이터로 목록을 채움 |

오늘은 SwiftData, 서버 통신, 앱 상태 관리를 설명하지 않습니다.
단순히 **여러 개의 글 데이터를 같은 모양으로 보여준다**는 점만 비교합니다.

## 샘플 글 정보 정리

아래 정보는 나중에 `src/data/posts.json`에 들어갈 JSON 초안입니다.

```json
[
  {
    "title": "",
    "slug": "",
    "date": "",
    "summary": "",
    "tags": []
  },
  {
    "title": "",
    "slug": "",
    "date": "",
    "summary": "",
    "tags": []
  }
]
```

템플릿은 [글 목록 샘플 데이터 템플릿](../templates/post-list-data.md)을 참고하세요.

## 공통 프롬프트

```text
Posts 페이지에 글 목록 기본 구조를 만들고 싶어.

현재 상태:
- Astro 개인 홈페이지가 있다.
- /posts 페이지가 있다.
- 공통 Layout과 Navigation이 있다.

목표:
- 샘플 글 1~2개를 보여준다.
- 각 글에는 title, slug, date, summary, tags가 보이게 한다.
- 글 정보는 `src/data/posts.json` 파일로 분리한다.
- 나중에 다른 콘텐츠 소스로 바꿔도 화면 코드를 크게 고치지 않는 구조로 만든다.
- 지금은 외부 API나 token, secret을 사용하지 않는다.
- 글 목록의 항목은 이번 수업의 기준이며, 나중에 바꿀 수 있다고 설명한다.
- Posts는 글 중심 콘텐츠라서 Projects 카드와 다른 구조로 다룬다고 설명한다.
- 나중에 MDX 같은 글 파일 방식이나 다른 콘텐츠 소스로 확장할 수 있다고 짧게 설명한다.
- 나중에 외부 API 응답을 연결하더라도, 응답 원본을 화면에 바로 넣지 말고 오늘의 `title`, `slug`, `date`, `summary`, `tags` 구조로 바꿔서 사용한다.

샘플 글 정보는 src/data/posts.json에 넣을 수 있는 아래 JSON 구조로 정리해줘.

[
  {
    "title": "첫 배포를 하며 배운 것",
    "slug": "first-deploy-reflection",
    "date": "2026-05-19",
    "summary": "Astro와 GitHub Pages로 개인 홈페이지를 처음 배포하며 배운 점을 정리합니다.",
    "tags": ["Astro", "GitHub Pages", "회고"]
  },
  {
    "title": "AI에게 코딩을 맡길 때 필요한 맥락",
    "slug": "ai-coding-context",
    "date": "2026-05-19",
    "summary": "좋은 프롬프트가 왜 필요한지, 계획과 검증을 어떻게 요청하는지 정리합니다.",
    "tags": ["Codex", "AI", "작업기록"]
  }
]

제약:
- 외부 API를 지금 붙이지 마.
- token, secret, API key를 만들거나 코드에 쓰지 마.
- 새 라이브러리를 추가하지 마.
- 관련 없는 파일은 수정하지 마.
- 실제 글 상세 페이지까지 만들 필요는 없어. 오늘은 목록 구조가 목표야.

작업 방식:
1. 먼저 현재 Posts 페이지를 확인해줘.
2. 샘플 글 정보를 `src/data/posts.json`에 두는 구조를 제안해줘.
3. 나중에 다른 콘텐츠 소스로 바꿀 때 어떤 부분만 교체하면 될지 설명해줘.
4. 외부 응답은 중첩 구조일 수 있으니, 화면에서 쓰기 쉬운 JSON 구조로 변환해서 사용한다고 설명해줘.
5. 가능하면 SwiftUI의 List나 ForEach와 비교해서 설명해줘.
6. 단, SwiftData, 서버 통신, 앱 상태 관리는 오늘 범위에서 빼줘.
7. 아직 수정하지 말고 계획만 보여줘.
8. 내가 승인하면 구현해줘.

완료 후 알려줄 것:
1. 변경한 파일 목록
2. 글 정보가 `src/data/posts.json`에 있는지
3. title, slug, date, summary, tags가 어디에 표시되는지
4. 나중에 다른 콘텐츠 소스로 바꿀 때 이 구조가 어떤 역할을 하는지
5. 브라우저 확인 주소
6. SwiftUI의 List나 ForEach와 비교했을 때 비슷한 점 한 줄
```

## 외부 연동을 미루는 프롬프트

AI가 너무 빨리 외부 API나 인증 설정을 붙이려고 하면 이렇게 말하세요.

```text
지금은 외부 콘텐츠 소스를 연결하는 단계가 아니야.
API, token, secret은 아직 사용하지 마.
오늘은 src/data/posts.json 파일의 샘플 글 정보로 Posts 목록 구조만 만들어줘.
나중에 다른 콘텐츠 소스로 바꿀 수 있도록, 어떤 부분을 교체하면 되는지만 설명해줘.
외부 응답 원본을 화면에 바로 쓰는 방식이 아니라, title, slug, date, summary, tags로 정리한 JSON을 화면에서 쓰는 방식으로 설명해줘.
```

## 중간 점검 질문

```text
Posts 구조를 나중에 다른 콘텐츠 소스로 바꿀 수 있는 이유를 쉽게 설명해줘.

아래를 나눠서 알려줘.
1. 지금은 `src/data/posts.json`의 어떤 샘플 글 정보를 쓰는지
2. 나중에 다른 콘텐츠 소스로 바뀔 부분은 어디인지
3. 오늘 일부러 만들지 않은 기능은 무엇인지
4. `slug`는 나중에 어떤 주소를 만들 때 쓰일 수 있는지
5. 지금 구조가 다음 확장 작업에 어떻게 이어지는지
```

## 체크박스

- [ ] Posts 페이지에 샘플 글이 보인다.
- [ ] 각 글에 제목이 보인다.
- [ ] 각 글에 주소용 `slug`가 있다.
- [ ] 날짜가 보인다.
- [ ] 요약이 보인다.
- [ ] 태그가 보인다.
- [ ] 외부 API를 붙이지 않았다.
- [ ] 나중에 바꿀 데이터 위치를 알고 있다.

## Ready Gate

다음 단계로 넘어가기 전에 아래가 되어 있어야 합니다.

- Posts 페이지가 단순 임시 제목만 있는 상태가 아닙니다.
- 샘플 글 목록이 보입니다.
- 외부 API 관련 secret이나 token을 만들지 않았습니다.
