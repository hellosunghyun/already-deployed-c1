# 11. 결과 기록과 다음 숙제

## 목표

2회차 결과와 막힌 지점을 기록합니다.

## 왜 하는가

2회차부터는 파일 구조가 많아집니다.
오늘 무엇을 만들었고 어디서 막혔는지 기록해두지 않으면 3회차에서 디자인 수정을 요청할 때 맥락이 부족해집니다.

## 오늘 마무리 기준

오늘 마무리 기준은 아래 순서입니다.

### 1순위. 현재 상태 기록

```text
GitHub 저장소 URL:
GitHub Pages URL 또는 없음:
열리는 페이지:
안 열리는 페이지:
build 결과:
오늘 막힌 지점:
```

### 2순위. 3회차 디자인 수정 재료

```text
다음 세션에서 고치고 싶은 화면:
고치고 싶은 이유:
참고하고 싶은 홈페이지:
가져오고 싶은 점:
따라하지 않을 점:
```

### 3순위. AGENTS.md 갱신

시간이 있으면 AGENTS.md를 현재 구조에 맞게 갱신합니다.
시간이 부족하면 갱신할 프롬프트만 기록해도 됩니다.

오늘 수업 중 반드시 파일까지 수정하지 않아도 됩니다.
다음 세션 전 숙제로 처리해도 됩니다.

- Projects:
- Project detail:
- Posts:

오늘 만든 commit:

다음 세션에서 디자인으로 고치고 싶은 것:

참고한 레퍼런스 사이트:
레퍼런스에서 가져오고 싶은 점:
- 레이아웃:

## AI에게 결과 요약 요청

```text
2회차 작업 결과를 다음 세션에서 이어갈 수 있게 요약해줘.

아래 형식으로 정리해줘.

1. 오늘 만든 페이지
2. 오늘 만든 공통 구조
3. Projects 카드 데이터 위치
4. Projects 상세 페이지 또는 상세 링크 상태
5. Posts 샘플 데이터 위치
6. build / deploy 확인 결과
7. 아직 부족한 점
8. 3회차에서 디자인 수정 요청을 할 때 참고할 포인트

중요:
- 과장하지 말고 실제 확인한 내용만 써줘.
- 실패한 것이 있으면 실패했다고 정확히 써줘.
- 다음에 내가 AI에게 이어서 보낼 수 있는 프롬프트도 하나 만들어줘.
```

## AGENTS.md 현재 구조 반영

1회차에서 만든 `AGENTS.md`는 처음에는 Home 첫 버전 기준이었습니다.
2회차가 끝나면 사이트 구조가 Home / About / Projects / Posts로 바뀌었으므로, 교육생이 자기 프로젝트의 현재 구조에 맞게 `AGENTS.md`도 한 번 갱신합니다.
AGENTS.md 갱신은 좋은 마무리 작업이지만, 오늘의 필수 완주 조건은 아닙니다.
시간이 있으면 실제 파일까지 수정하고, 시간이 부족하면 아래 프롬프트를 기록해두는 것으로 충분합니다.

아래 프롬프트를 그대로 보내도 됩니다.

```text
현재 프로젝트 구조에 맞게 AGENTS.md를 업데이트하고 싶어.

먼저 확인해줘:
1. 현재 AGENTS.md 내용
2. src/pages 안의 페이지 구조
3. Layout / Navigation 파일이 있다면 위치
4. Projects 데이터 파일이 있다면 위치
5. Posts 데이터 파일이 있다면 위치

반영하고 싶은 현재 구조:
- 이 프로젝트는 Astro 기반 개인 홈페이지다.
- 현재 사이트는 Home / About / Projects / Posts 구조를 가진다.
- Projects는 카드와 상세 페이지 또는 상세 링크를 가진다.
- Projects 정보는 가능하면 src/data/projects.json 같은 JSON 파일에서 관리한다.
- Posts 정보는 가능하면 src/data/posts.json 같은 JSON 파일에서 관리한다.
- 나중에 외부 콘텐츠 소스를 붙이더라도, API 응답 원본을 화면에 바로 쓰지 말고 화면에서 쓰기 쉬운 JSON 구조로 변환해서 사용한다.
- Codex에서는 Browser로 로컬 화면을 확인한다.
- 수정 후에는 브라우저 확인과 npm run build를 우선한다.
- secret, token, password, API key는 코드에 쓰지 않는다.
- commit과 push는 변경사항 요약을 먼저 보여주고 사용자 승인 후 진행한다.

작업 방식:
- 아직 AGENTS.md를 수정하지 마.
- 먼저 현재 파일과 비교해서 어떤 문장을 바꾸거나 추가할지 제안해줘.
- 너무 긴 규칙 파일로 만들지 말고, 앞으로 작업할 때 꼭 필요한 규칙만 남겨줘.
- 내가 승인하면 AGENTS.md를 수정해줘.

완료 후 알려줄 것:
1. AGENTS.md에서 바뀐 내용 요약
2. 현재 프로젝트 구조가 어떻게 반영됐는지
3. 다음 세션에서 AI가 특히 기억해야 할 규칙 3개
```

승인 후에는 이렇게 말합니다.

```text
좋아. 방금 제안한 방향으로 AGENTS.md를 수정해줘.
수정 후 변경 내용을 짧게 요약하고, npm run build가 필요한 변경인지도 판단해줘.
```

## 다음 세션 전 숙제

3회차에서는 브라우저 화면과 구체적인 요구사항을 바탕으로 디자인을 수정합니다.

아래를 준비해오면 훨씬 수월합니다.

- 현재 내 홈페이지 스크린샷 1장
- 마음에 드는 개인 홈페이지 주소 또는 레퍼런스 링크 2~5개
- 고치고 싶은 부분 3개
- 모바일에서 이상한 부분 1개
- 유지하고 싶은 부분 1개

레퍼런스는 다양할수록 좋습니다.
다만 3회차에서 모든 요소를 한 번에 적용하기는 어렵기 때문에, 여러 예시를 본 뒤 지금 내 홈페이지에 가장 쉽게 적용할 수 있는 요소를 고릅니다.

개인 홈페이지 주소를 직접 찾아와도 좋고, 아래 같은 레퍼런스 사이트에서 마음에 드는 예시를 골라도 됩니다.

| 레퍼런스 사이트 | 볼 만한 기준 |
|---|---|
| [Siteinspire - Personal](https://www.siteinspire.com/websites/category/personal) | 개인 홈페이지와 포트폴리오 구조 |
| [Land-book](https://land-book.com/) | 최신 웹사이트의 첫 화면, 섹션 구성 |
| [Awwwards](https://www.awwwards.com/websites/) | 강한 비주얼, 인터랙션, 실험적인 표현 |
| [One Page Love](https://onepagelove.com/inspiration/) | 한 페이지로 정리된 홈페이지 구조 |
| [Lapa Ninja](https://www.lapa.ninja/) | 랜딩 페이지와 섹션별 구성 |
| [Godly.design](https://godly.design/) | 실제 서비스 화면, UI 패턴 |
| [Minimal Gallery](https://minimal.gallery/) | 단순하고 정돈된 미니멀 웹사이트 |

레퍼런스를 볼 때는 "전체를 똑같이 만들기"가 아니라 아래처럼 한두 가지 포인트만 고릅니다.
기준은 멋진 것보다 **이번 홈페이지에 바로 적용하기 쉬운 것**입니다.

```text
이 사이트에서 가져오고 싶은 점:
- 첫 화면의 정보 순서
- About 문장의 길이
- Projects 카드 간격
- 글 목록의 밀도
- 모바일에서 제목이 보이는 방식
- 지금 내 홈페이지에 바로 적용하기 쉬운 점

따라하지 않을 점:
- 너무 무거운 애니메이션
- 읽기 어려운 작은 글자
- 내 콘텐츠와 맞지 않는 과한 분위기
```

## 3회차 예고

3회차에서는 아래 요청을 더 잘하는 법을 다룹니다.

또 Codex Browser에서 화면의 특정 요소나 영역을 지정하고, 그 대상을 기준으로 프롬프트를 더 정확하게 쓰는 방법도 다룹니다.

예를 들어 전체 페이지를 막연히 고치라고 하기보다 아래처럼 요청합니다.

```text
브라우저 화면에서 선택한 Projects 카드 영역을 기준으로 봐줘.
카드 사이 간격은 유지하되, 제목과 설명의 위계가 더 분명해지게 수정 계획을 먼저 제안해줘.
```

나쁜 요청:

```text
좀 예쁘게 해줘.
```

좋은 요청:

```text
Home 첫 화면의 제목과 소개 문장 사이 간격이 너무 넓어 보여.
모바일에서는 제목이 3줄로 길게 깨져.
전체 톤은 유지하되, 제목 크기와 섹션 간격을 줄이는 방향으로 먼저 계획을 제안해줘.
```

## 체크박스

- [ ] GitHub 저장소 URL을 기록했다.
- [ ] GitHub Pages URL을 기록했다. 또는 실패 로그를 기록했다.
- [ ] 오늘 완료한 페이지를 표시했다.
- [ ] 오늘 만든 commit을 기록했다.
- [ ] 열린 URL과 안 열린 URL을 구분해 적었다.
- [ ] Projects / Posts가 어디까지 됐는지 적었다.
- [ ] build 성공 결과 또는 실패 로그를 기록했다.
- [ ] 현재 구조에 맞게 AGENTS.md 업데이트 프롬프트를 입력했다.
- [ ] 다음 세션에서 고치고 싶은 부분을 적었다.
- [ ] 참고하고 싶은 개인 홈페이지 주소 또는 레퍼런스 링크를 2~5개 찾았다.
- [ ] 레퍼런스에서 가져오고 싶은 점과 따라하지 않을 점을 각각 적었다.

## Ready Gate

오늘 마무리 전에 아래가 되어 있으면 충분합니다.

- 2회차 결과가 기록되어 있습니다.
- 열린 URL과 안 열린 URL을 알고 있습니다.
- build 성공 결과 또는 실패 로그가 있습니다.
- Projects / Posts는 완성 여부와 현재 상태가 기록되어 있습니다.
- `AGENTS.md`가 현재 구조에 맞게 갱신되었거나, 갱신할 프롬프트가 기록되어 있습니다.
- GitHub Pages가 열리거나, 실패 로그가 있습니다.
- 3회차에서 디자인 수정을 요청할 재료가 있습니다.
