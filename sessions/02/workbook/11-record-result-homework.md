# 11. 결과 기록과 다음 숙제

## 목표

2회차 결과와 막힌 지점을 기록합니다.

## 왜 하는가

2회차부터는 파일 구조가 많아집니다.
오늘 무엇을 만들었고 어디서 막혔는지 기록해두지 않으면 3회차에서 디자인 수정을 요청할 때 맥락이 부족해집니다.

## 결과 기록 폼

아래 값을 기록합니다.

```text
GitHub 저장소 URL:
GitHub Pages URL:
오늘 완료한 페이지:
- Home:
- About:
- Projects:
- Posts:

오늘 만든 commit:
오늘 막힌 지점:
다음 세션에서 디자인으로 고치고 싶은 것:
마음에 드는 참고 홈페이지 URL:
```

## AI에게 결과 요약 요청

```text
2회차 작업 결과를 다음 세션에서 이어갈 수 있게 요약해줘.

아래 형식으로 정리해줘.

1. 오늘 만든 페이지
2. 오늘 만든 공통 구조
3. Projects 카드 데이터 위치
4. Posts 샘플 데이터 위치
5. build / deploy 확인 결과
6. 아직 부족한 점
7. 3회차에서 디자인 수정 요청을 할 때 참고할 포인트

중요:
- 과장하지 말고 실제 확인한 내용만 써줘.
- 실패한 것이 있으면 실패했다고 정확히 써줘.
- 다음에 내가 AI에게 이어서 보낼 수 있는 프롬프트도 하나 만들어줘.
```

## 다음 세션 전 숙제

3회차에서는 브라우저 화면과 구체적인 요구사항을 바탕으로 디자인을 수정합니다.

아래를 준비해오면 훨씬 수월합니다.

- 현재 내 홈페이지 스크린샷 1장
- 마음에 드는 개인 홈페이지 링크 1~2개
- 고치고 싶은 부분 3개
- 모바일에서 이상한 부분 1개
- 유지하고 싶은 부분 1개

## 3회차 예고

3회차에서는 아래 요청을 더 잘하는 법을 다룹니다.

나쁜 요청:

```text
좀 예쁘게 해줘.
```

좋은 요청:

```text
Home 첫 화면의 제목과 소개 문장 사이 간격이 너무 넓어 보여.
모바일에서는 제목이 3줄로 길게 깨져.
전체 톤은 유지하되, 제목 크기와 섹션 간격을 줄이는 방향으로 먼저 계획을 제안해줘.
```

## 체크박스

- [ ] GitHub 저장소 URL을 기록했다.
- [ ] GitHub Pages URL을 기록했다. 또는 실패 로그를 기록했다.
- [ ] 오늘 완료한 페이지를 표시했다.
- [ ] 오늘 만든 commit을 기록했다.
- [ ] 다음 세션에서 고치고 싶은 부분을 적었다.
- [ ] 참고하고 싶은 홈페이지 링크를 하나 찾았다.

## Ready Gate

오늘 마무리 전에 아래가 되어 있으면 충분합니다.

- 2회차 결과가 기록되어 있습니다.
- GitHub Pages가 열리거나, 실패 로그가 있습니다.
- 3회차에서 디자인 수정을 요청할 재료가 있습니다.

## 마무리 문장

오늘 만든 것은 완성된 포트폴리오가 아닙니다.
하지만 이제 사이트에는 방문자가 움직일 수 있는 길이 생겼습니다.

```text
1회차: 인터넷에 올라가는 첫 버전
2회차: 읽히는 정보 구조
3회차: 화면을 보며 디자인 수정
4회차: Notion 콘텐츠 연결
5회차: 운영 가능한 홈페이지로 정리
```
