# 07. Projects 카드 만들기

## 목표

Projects 페이지에 프로젝트 또는 경험 3개를 카드 형태로 보여주고, 카드를 클릭하면 상세정보로 이동할 수 있게 합니다.

## 왜 하는가

Projects는 "나는 이런 것을 해봤다"를 보여주는 증거입니다.
단순히 이름만 나열하기보다, 무엇을 했고 무엇을 배웠는지 구조화하면 방문자가 이해하기 쉽습니다.
Projects를 About과 따로 두는 이유는 비슷한 형식의 사례가 반복되기 때문입니다.
목록에서는 핵심만 보여주고, 관심 있는 카드를 누르면 더 자세한 배경과 과정으로 들어가게 만들 수 있습니다.

## 쉬운 설명

좋은 프로젝트 카드에는 아래 정보가 들어갑니다.

이 기준은 이번 수업에서 프로젝트를 쉽게 정리하기 위한 기본값입니다.
나중에는 이미지, 링크, 긴 회고, 짧은 실험 기록처럼 다른 형식으로 바꿔도 됩니다.

- 프로젝트 이름
- 한 줄 설명
- 내가 맡은 역할
- 사용한 도구 또는 기술
- 결과 또는 배운 점
- 자세히 보기 링크

반복되는 카드는 Component로 만들면 좋습니다.
프로젝트 내용은 `src/data/projects.json` 같은 JSON 파일에 따로 두면 나중에 추가하기 쉽습니다.
카드에는 상세 페이지 주소를 직접 넣기보다 `slug`를 넣어두면 좋습니다.
예를 들어 `first-homepage`라는 slug가 있으면 화면 코드에서 `/projects/first-homepage` 주소를 만들 수 있습니다.

## 아주 쉽게 말하면

Projects 카드는 "같은 양식의 소개 카드"입니다.

```text
카드 양식:
- 제목
- 설명
- 역할
- 도구
- 결과
- 자세히 보기
```

프로젝트가 3개라면 양식을 3번 새로 만드는 것이 아니라, 같은 양식에 내용만 바꿔 넣습니다.

```text
ProjectCard = 카드 양식
src/data/projects.json = 카드에 넣을 내용 목록
detail page = 카드에서 들어가는 상세 설명
```

JSON이 어렵다면 오늘은 **여러 프로젝트 정보를 한 파일에 모아둔 표준적인 정보 파일**이라고 생각해도 됩니다.

## SwiftUI와 비교해서 이해하기

SwiftUI에서 비슷한 프로젝트 카드를 만든다면 보통 아래처럼 생각할 수 있습니다.

| Astro | SwiftUI에서 떠올릴 쉬운 개념 |
|---|---|
| `ProjectCard.astro` | `ProjectCardView` |
| `src/data/projects.json` | 프로젝트 목록 데이터 |
| JSON 목록을 돌며 카드 표시 | `ForEach` |
| `title`, `description` 같은 값 | 카드에 넣을 데이터 |
| 자세히 보기 링크 | 카드를 눌렀을 때 이동할 화면 |

핵심은 Swift 문법이나 JSON 문법을 외우는 것이 아닙니다.
**프로젝트 정보 파일**과 **카드를 보여주는 모양**을 나누면, 프로젝트가 늘어날 때 수정하기 쉽다는 점을 이해하면 됩니다.

## 시간이 부족할 때의 최소 산출물

Projects 카드를 화면에 완성하지 못해도 아래 중 하나는 남깁니다.

```text
1. 프로젝트 정보 1~3개 초안
2. src/data/projects.json에 넣을 수 있는 JSON 초안
3. 카드에 들어갈 항목 목록
4. 상세 페이지가 안 열리는 경우 어떤 slug 또는 URL에서 막혔는지
```

오늘 Projects의 목표는 완성된 포트폴리오 문장을 쓰는 것이 아닙니다.
반복되는 카드 구조와 데이터 분리 방식을 이해하는 것입니다.

## 공통 프롬프트

이 프롬프트는 일부러 자세합니다.
길다고 줄이기보다, 현재 상태와 확인 기준을 빠뜨리지 않기 위해 그대로 복사해서 사용하세요.

```text
Projects 페이지에 프로젝트 카드 3개를 추가하고 싶어.

현재 상태:
- Astro 개인 홈페이지가 있다.
- /projects 페이지가 있다.
- 공통 Layout과 Navigation이 있다.

내 프로젝트 정보는 src/data/projects.json에 넣을 수 있는 아래 JSON 구조로 정리해줘.

[
  {
    "title": "입력",
    "description": "입력",
    "role": "입력",
    "tools": ["입력"],
    "result": "입력",
    "detail": "프로젝트 배경, 과정, 배운 점을 3~5문장으로 입력",
    "slug": "first-homepage"
  },
  {
    "title": "입력",
    "description": "입력",
    "role": "입력",
    "tools": ["입력"],
    "result": "입력",
    "detail": "입력",
    "slug": "project-two"
  },
  {
    "title": "입력",
    "description": "입력",
    "role": "입력",
    "tools": ["입력"],
    "result": "입력",
    "detail": "입력",
    "slug": "project-three"
  }
]

목표:
- Projects 페이지에 카드 3개를 보여준다.
- 각 카드에 title, description, role, tools, result가 보이게 한다.
- 각 카드는 `slug`를 이용해 해당 프로젝트 상세정보로 이동하게 한다.
- 상세정보에는 프로젝트 배경, 내가 한 일, 사용 도구, 결과 또는 배운 점이 보이게 한다.
- 프로젝트 데이터와 카드 화면을 분리한다.
- 같은 카드 구조를 반복해서 사용한다.
- 나중에 프로젝트를 쉽게 추가할 수 있게 한다.
- 카드에 들어갈 항목은 이번 수업의 기준이며, 정답처럼 고정하지 않는다.

구현 방향:
- 가능하면 ProjectCard 같은 작은 Component를 만든다.
- 프로젝트 목록은 `src/data/projects.json` 파일로 관리한다.
- 카드에는 `slug`를 두고, 화면 코드에서 `/projects/{slug}` 형태의 상세 페이지 주소를 만든다.
- 상세 페이지는 오늘 범위에서는 복잡한 동적 라우팅보다 정적 페이지로 단순하게 만들어도 된다.
- 데이터가 많아져도 카드 코드 자체를 계속 복사하지 않게 한다.
- 나중에 다른 콘텐츠 소스로 바꾸더라도 페이지와 Component를 크게 고치지 않도록, 화면 코드와 정보 파일을 분리한다.
- 나중에 외부 API 응답을 연결하더라도, 응답 원본을 화면에 바로 넣지 말고 오늘의 `title`, `description`, `role`, `tools`, `result`, `detail`, `slug` 구조로 바꿔서 사용한다.

제약:
- 새 라이브러리를 추가하지 마.
- 관련 없는 파일은 수정하지 마.
- 디자인은 기존 Layout과 어울리게만 정리해줘.
- 너무 화려한 카드 디자인은 피하고 읽기 쉽게 해줘.
- 외부 이미지나 아이콘은 지금 추가하지 마.

작업 방식:
1. 먼저 현재 Projects 페이지 구조를 확인해줘.
2. 만들 파일과 수정할 파일을 제안해줘.
3. 카드 클릭 후 상세정보를 보여줄 방식을 제안해줘.
4. `src/data/projects.json`과 ProjectCard Component가 각각 어떤 역할인지 초보자도 이해할 수 있게 설명해줘.
5. 가능하면 SwiftUI의 ProjectCardView, 목록 데이터, ForEach, 화면 이동 정도와 비교해서 설명해줘.
6. 상세 페이지를 만들 때는 오늘 범위에서 이해 가능한 단순한 정적 페이지 방식으로 제안해줘.
7. 단, Identifiable, 상태 관리, 복잡한 데이터 흐름, 동적 라우팅 심화는 오늘 범위에서 빼줘.
8. 아직 수정하지 말고 계획만 보여줘.
9. 내가 승인하면 구현해줘.

완료 후 알려줄 것:
1. 변경한 파일 목록
2. 프로젝트 정보가 `src/data/projects.json`에 있는지
3. 카드 Component가 있다면 그 역할
4. 프로젝트를 하나 더 추가하려면 어디를 수정하면 되는지
5. 각 카드의 `slug`가 어떤 상세 페이지 주소로 연결되는지
6. 브라우저 확인 주소
7. SwiftUI의 ForEach와 화면 이동을 비교했을 때 비슷한 점 한 줄
```

## 프로젝트가 아직 없을 때 프롬프트

```text
아직 보여줄 프로젝트가 3개 없어서 임시 카드로 시작하고 싶어.

조건:
- 실제 경력처럼 과장하지 마.
- "준비 중", "학습 중", "워크숍 실습"처럼 솔직한 표현을 사용해줘.
- 나중에 쉽게 바꿀 수 있게 데이터 위치를 명확히 해줘.
- 프로젝트 정보는 가능하면 `src/data/projects.json`에 모아줘.
- 카드 예시는 개인 홈페이지에 올려도 부끄럽지 않을 정도로 담백하게 써줘.
- 카드 클릭 시 볼 수 있는 상세 설명도 짧게 만들어줘.
```

## 상세정보까지 연결하는 프롬프트

카드 목록이 만들어진 뒤, 상세정보 연결을 별도로 요청하고 싶다면 이렇게 말합니다.

```text
방금 만든 Projects 카드가 클릭 가능하도록 상세 페이지를 연결하고 싶어.

조건:
- 각 카드에 자세히 보기 링크를 넣어줘.
- 각 프로젝트별 상세 페이지를 단순한 정적 페이지로 만들어줘.
- 상세 페이지에는 프로젝트 배경, 내가 한 일, 사용 도구, 결과 또는 배운 점을 넣어줘.
- 프로젝트 정보는 `src/data/projects.json`에서 관리하게 해줘.
- 프로젝트 정보에는 내부 주소 전체를 직접 넣지 말고 `slug`를 두고, 화면 코드에서 `/projects/{slug}` 주소를 만들게 해줘.
- 나중에 다른 콘텐츠 소스로 바꾸기 쉽도록, 화면 코드에 프로젝트 문장을 직접 흩뿌리지 마.
- 나중에 외부 API 응답을 붙이더라도, 원본 응답을 화면에 바로 쓰지 않고 오늘의 JSON 구조로 변환해서 쓰는 방식으로 설명해줘.
- 동적 라우팅이나 복잡한 데이터 구조는 오늘 범위에서 빼줘.
- 기존 Projects 카드 디자인은 크게 바꾸지 마.
- 수정 전에는 만들 파일과 수정할 파일을 먼저 제안해줘.
```

## 중간 점검 질문

```text
Projects 카드 구조를 초보자도 이해할 수 있게 다시 설명해줘.

아래를 꼭 포함해줘.
1. 프로젝트 정보가 `src/data/projects.json`에 있는지
2. 카드 모양을 담당하는 코드는 어디에 있는지
3. 카드 클릭 시 어느 상세 페이지로 이동하는지
4. 프로젝트를 하나 더 추가하려면 어디를 바꾸면 되는지
5. `slug`가 왜 내부 상세 페이지 주소를 만들 때 필요한지
6. 오늘 꼭 이해해야 하는 것과 아직 몰라도 되는 것을 나눠서 설명해줘.
```

## 체크박스

- [ ] Projects 페이지에 카드 3개가 보인다.
- [ ] 각 카드에 제목이 있다.
- [ ] 각 카드에 한 줄 설명이 있다.
- [ ] 각 카드에 내 역할이 있다.
- [ ] 각 카드에 사용 도구 또는 기술이 있다.
- [ ] 각 카드에 결과 또는 배운 점이 있다.
- [ ] 각 카드를 클릭하면 상세정보로 이동할 수 있다.
- [ ] 상세 페이지에 프로젝트 배경과 배운 점이 있다.
- [ ] 프로젝트를 하나 더 추가할 위치를 안다.

## Ready Gate

다음 단계로 넘어가기 전에 아래 중 최소한 하나는 남겨야 합니다.

- Projects 페이지에 카드가 보입니다.
- 아직 카드가 안 보인다면 프로젝트 정보 JSON 초안이 있습니다.
- 상세 페이지가 안 열리면 어떤 `slug` 또는 URL에서 막혔는지 기록했습니다.
- 프로젝트를 하나 더 추가하려면 어디를 바꾸면 되는지 알고 있습니다.
